# Ledger Tower Progress

Last updated: `2026-06-01`

## Current State

- `Ledger Tower` is the first companion environment under `envs\ledger_tower`.
- It is a deterministic fixed-value tower puzzle RPG for exact HP/key/gold/stat route accounting.
- The fixed core map is unchanged; current work split track semantics for practice and official attempts, added an explicit `boss_gated` variant for forced-combat routing, and promoted one generated 8-floor `shop_timing` candidate as an extended variant track.
- Current default map topology allows no-combat victory; combat-required benchmarks should use an explicit boss/enemy goal or the separate `boss_gated` variant map.
- Track semantics now distinguish:
  - whitebox direct SDK practice,
  - server-token practice budgets,
  - no-practice token-limited official runs,
  - multi-official-attempt best-of-N scoring.

## Mechanics

- The default/core map routes through six compact floors; extended generated variants can use alternate floor counts.
- Important ledgers are HP, attack, defense, gold, yellow keys, blue keys, current position, opened doors, defeated monsters, and inventory.
- Combat is deterministic and player-first. `preview` exposes deterministic fight cost; fights that would drop HP to `0` or less are rejected without spending a move.
- Shops sell attack, defense, and HP. Purchases are intentionally route-shaping.
- Verification uses move budget as the primary hard budget and soft-stop move as an advisory scoring signal.
- High-score tracks with soft stop set `soft_stop_scoring: linear_to_hard_budget`, so `softStopScore` decays from `1` at the soft stop to `0` at the hard budget instead of dropping to `0` on the first overrun. Non-migrated tracks keep the shared default `binary` behavior. The `boss-gated-high-score` and `generated-shop-timing-8f-high-score` variants explicitly disable soft stop and rely on hard budget plus route score for quality.
- `route_score` rewards clear success, retained resources, and efficient movement; remaining keys are intentionally low-weight tie-breakers so they do not dominate route quality.
- High-score movement tax is now `-30` per move. Historical runs before this change used `-15` per move; for successful old samples, the new comparable route score is `old_route_score - 15 * moves`.
- Current default map topology does not force combat. A no-combat route can collect `ledger_core` and exit while bypassing `f6_boss`, so boss/combat requirements must be explicit track goals or variant-map constraints.
- The `boss_gated` variant keeps open routing zones on each floor but places a boss on the only passable cut point between the entry-side zone and the upward/exit-side zone.
- The `boss_gated` variant now has explicit optional route-value branches beyond the mandatory boss cuts: a low-damage f1 blue-key detour that can support the f4 blue shortcut, a f3 duelist/sharp-ledger attack branch, a f5 sentinel/large-potion HP-gold branch, and a f6 ward-gem defense branch. The final Balance Auditor is also harder, so the old 54-move attack-only route no longer clears.

## Tracks

- `tutorial-clear`: short mechanics track; target is reaching `f3`.
- `ledger-clear`: full clear with victory, boss, `ledger_core`, and HP floor.
- `high-score`: legacy whitebox high-score track.
- `high-score-whitebox`: direct SDK high-score variant with explicit `route_score` and verification reward formula disclosure.
- `high-score-token-limited`: server-only, no practice token, one official attempt.
- `high-score-practice-budgeted`: server-only, three non-official practice games, one official attempt.
- `high-score-best-of-3`: server-only, three official attempts, runner selects best verified reward.
- `boss-gated-high-score`: whitebox high-score track on the `boss_gated` variant; every floor boss is structurally unavoidable.
- `generated-shop-timing-8f-high-score`: extended whitebox high-score track on the promoted 8-floor generated `shop_timing` variant.
- `core` suite now uses `tutorial-clear`, `ledger-clear`, and `high-score-token-limited`.
- `extended` contains disclosure, practice-budget, best-of-N, boss-gated, and promoted generated-map tracks; `scoring` compares the fixed-map high-score variants.

## Token And Scoring Semantics

- Server tokens now record `token_role`, `official`, and `scoring`.
- Practice-token games are non-official and cannot be verified.
- Official tokens carry max-new-game count and scoring policy.
- Official tokens can carry `soft_stop_scoring`; default is `binary`, while Ledger Tower high-score tracks use `linear_to_hard_budget` when soft stop is enabled. Tracks can set `soft_stop: false` to mint official tokens without a soft-stop tick.
- Official verification is now a submission boundary: after `verify()`, that
  official game is closed to further commands and another scored sample must use
  a fresh official game.
- Official attempt count is based on official game creation, not active open
  games; `verify()` never refunds, frees, or restores a consumed attempt.
- Runner summaries select best-of-N verification when `official_attempts > 1`.
- Selection order is highest verification reward first, then accepted/success outcome and route-quality tie breakers.
- Ledger server SDK supports `token_role="practice"` and reads `LEDGER_TOWER_PRACTICE_AUTH_TOKEN` automatically.
- Runners inject `LEDGER_TOWER_SERVER_URL`, `LEDGER_TOWER_AUTH_TOKEN`, and, when needed, `LEDGER_TOWER_PRACTICE_AUTH_TOKEN`.
- Variant tracks can record `data_path` on official/practice tokens and inject `LEDGER_TOWER_DATA_PATH` for direct SDK practice, keeping direct and official sessions on the same selected map.
